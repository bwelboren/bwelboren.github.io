CMSpit Walkthrough

Questions

  • What is the name of the Content Management System (CMS) installed on the server?
  • What is the version of the Content Management System (CMS) installed on the server?
  • What is the path that allow user enumeration?
  • How many users can you identify when you reproduce the user enumeration attack? What is the path that allows you to change user account passwords?
  • Compromise the Content Management System (CMS). What is Skidy's email.
  • What is the web flag?
  • Compromise the machine and enumerate collections in the document database installed in the server. What is the flag in the database?
  • What is the user.txt flag?
  • What is the CVE number for the vulnerability affecting the binary assigned to the system user? Answer format: CVE-0000-0000
  • What is the utility used to create the PoC file?
  • Escalate your privileges. What is the flag in root.txt?

Approach

Scanning the IP I found an open port 80. Upon visiting the website we are immediately greeted with the name of the CMS.

While first checking the page itself for the version number we find nothing so we check the source of the page, thats where we find the version number of the CMS.

Using the CMS and the version number is usually the way to search for an exploit so we find two links of our interest

https://swarm.ptsecurity.com/rce-cockpit-cms/

https://packetstormsecurity.com/files/162282/Cockpit-CMS-0.11.1-NoSQL-Injection-Remote-Command-Execution.html

https://blog.convisoappsec.com/en/a-case-study-on-cve-2021-22204-exiftool-rce/

In the first link he explains what the exploit does and it reveals the paths it uses for the exploit, the second link shows us that there is a metasploit module available for the exploit.

On the third link scroll down to 'The Exploit' to learn with what tool/utility the PoC gets created

Booting up msfconsole we adjust the options and run it against the server, revealing usernames. While we look for the options of the exploit we also see an option called User for account takeover. We use the user that stands out to us and run the exploit again it will output the password that has been reset, now you can login.

Once logged in you can click on the icon left upper corner to get an overview of options available in the CMS admin panel, we see files, accounts.. etc. When we click on accounts we can get the emails of users.

We then navigate to the files to find anything of use and see one particular file that stands out to us. We also see that we can upload files, maybe a reverse shell?

We upload a reverse shell from pentestmonkey onto the site and run a listener then navigate to the location of the php reverse shell to start an interactive shell.

In the questions it states to get information from a database, if I remember correctly I found MongoDB on either the source of the website or in the nmap scan, so we see if we can run mongo with the command mongo and it works.

Now me being new to mongo I read the documentation and see if we can show any databases. We can with show dbs we see a database name with sudo backup in it and we use the database like so use db once inside the database we show collections and see flag and user.

It took me quiet a while to find how to access these collections so I landed on a website about pentesting mongodb from hacktricks which showed how to dump a collection with the following command.

db.flag.find() and we got another flag. Lets also see whats inside the other collection and we find credentials of a user.

Its time to exit mongo with exit and upgrade our shell with python3 -c 'import pty; pty.spawn("/bin/bash")' so we can login to other users. We login with the credentials and get the user flag in /home/stux/

Now we need to elevate the privileges of this user so usually the first thing I do in a system is run sudo -l which shows binaries we can use as other users.

It shows that we can run exiftool as root without password, visiting gtfobins and searching on the binary we scroll down to Sudo and see that this tool can be used to read and write files. In the case of the website Sudo is being used to write files so we adjust the one liner to: We change it to read from a file and output it to a directory we can read from.

sudo exiftool -filename=/root/root.txt /home/stux/root.txt

Now we got the rootflag

cat /home/stux/root.txt
